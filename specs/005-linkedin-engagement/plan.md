# Implementation Plan: LinkedIn Advanced Engagement

**Branch**: `005-linkedin-engagement` | **Date**: 2026-03-26 | **Spec**: [spec.md](spec.md)

## Summary

Build a `silver-linkedin-watcher` that detects new comments on own posts, AI-relevant feed posts, and AI-niche people — creating vault task files that flow through the existing HITL approval pipeline (cloud drafts → human approves → local agent executes via Playwright). Rate limiting, session locking, and deduplication protect the LinkedIn account from detection and restriction.

---

## Technical Context

**Language/Version**: Python 3.11+ (matches existing codebase)
**Primary Dependencies**: Playwright (sync API), python-dotenv, PyYAML, openai — all already installed
**Storage**: Flat files in vault + JSON state file + flat text deduplication files
**Testing**: Manual end-to-end test per user story; PM2 log inspection
**Target Platform**: Windows 11 local machine (same as all other watchers)
**Performance Goals**: Watcher poll cycle completes in < 3 minutes; cloud draft in < 30s; local execution in < 90s per action
**Constraints**: Max 10 LinkedIn comments/day, max 5 connections/day; single browser session at a time; 3–8s human-like delays
**Scale/Scope**: Single LinkedIn account; 3 engagement action types; ~15 tasks/day at full rate

## Implementation Phases

### Phase 1 — Foundation (P1: Reply to Comments)

**Goal**: End-to-end reply flow working. Safest feature first.

1. **`linkedin_rate_limiter.py`** — Shared utility
   - `RateLimiter` class: load/save state JSON, `can_execute(action_type)`, `record_action(action_type)`, `pause_account(reason)`, `is_paused()`
   - `SessionLock` class: `acquire(timeout=60)`, `release()`, context manager support

2. **`SilverTier/scripts/watchers/linkedin.py`** — Watcher (notification polling only for Phase 1)
   - Poll `linkedin.com/notifications/` every 120s (slower than Gmail — LinkedIn is stricter)
   - JS evaluate to extract comment notifications
   - Create `LINKEDIN_REPLY_*.md` in `Needs_Action/`
   - Deduplication via `processed_linkedin_comments.txt`
   - Security challenge detection → `pause_account()`

3. **`cloud_linkedin_reply_handler.py`**
   - Read task frontmatter (commenter_name, comment_snippet, post_url)
   - Build prompt: reply to `commenter_name`'s comment `comment_snippet` on a post about [brand context from Business_Goals.md]
   - OpenAI draft (max_tokens=200, temp=0.7)
   - Write `APPROVE_REPLY_LINKEDIN_*.md` to `Pending_Approval/`
   - Move task to `Plans/`

4. **`local_linkedin_reply_handler.py`**
   - Check `is_paused()` + `can_execute("reply")` (replies not rate-limited, only comments/connects)
   - Acquire session lock
   - Claim file to `In_Progress/local/`
   - Navigate to `post_url`, find comment by commenter_name, click Reply, type, submit
   - Release lock in `finally`
   - Audit log on success/failure

5. **Route wiring** — add `linkedin_reply` to `cloud_agent.py` and `linkedin_reply_approval` to `local_agent.py`

6. **`ecosystem.config.js`** — add `silver-linkedin-watcher` entry

---

### Phase 2 — Feed Scanning (P2: Comment on Others' Posts)

**Goal**: Feed scanning + commenting flow working.

1. **Extend `linkedin.py` watcher** — add feed scan task
   - Navigate to `linkedin.com/feed/`
   - JS evaluate to extract post cards: URL, author, text
   - Keyword filter against `LINKEDIN_KEYWORDS` env var
   - Max 5 posts per cycle
   - Deduplication via `processed_linkedin_posts.txt`
   - Rate-limit check before creating tasks (stop at 10 total comment tasks queued/day)

2. **`cloud_linkedin_comment_handler.py`**
   - Read post_author, post_snippet, post_url
   - Build prompt: write a specific, insightful comment on this post (include post text; ban generic phrases)
   - Max_tokens=150, temp=0.75
   - Write `APPROVE_COMMENT_LINKEDIN_*.md` to `Pending_Approval/`

3. **`local_linkedin_comment_handler.py`**
   - `can_execute("comment")` → check daily counter vs limit (10)
   - Acquire session lock
   - Claim file, navigate to post_url, click comment field, type, submit
   - `record_action("comment")` on success
   - Release lock

4. Route wiring in cloud/local agents

---

### Phase 3 — People Search (P3: Auto-Connect)

**Goal**: People discovery + connection request flow working.

1. **Extend `linkedin.py` watcher** — add people search task
   - Navigate to search URL with `LINKEDIN_CONNECT_KEYWORDS` env var
   - JS evaluate to extract profile cards: name, headline, company, profile URL
   - Filter: must be 2nd degree, must have AI-related headline keyword
   - Max 3 candidates per cycle
   - Deduplication via `processed_linkedin_profiles.txt`

2. **`cloud_linkedin_connect_handler.py`**
   - Read candidate_name, candidate_headline, candidate_company, profile_url
   - Build prompt: personalised connection note referencing candidate's specific role/company
   - Max_tokens=80 (≤300 chars), temp=0.6
   - Validate length ≤ 300 chars before writing approval
   - Write `APPROVE_CONNECT_LINKEDIN_*.md` to `Pending_Approval/`

3. **`local_linkedin_connect_handler.py`**
   - `can_execute("connect")` → check daily counter vs limit (5)
   - Acquire session lock
   - Claim file, navigate to profile_url, click Connect → Add a note → type note → Send
   - `record_action("connect")` on success
   - Release lock

4. Route wiring in cloud/local agents

## Safety Architecture Summary

```
linkedin_rate_limiter.py
├── RateLimiter
│   ├── load(): read linkedin_rate_state.json (create with defaults if missing)
│   ├── can_execute(type): check date reset + counter vs limit + account_paused
│   ├── record_action(type): increment counter + write atomically
│   └── pause_account(reason): set account_paused=true (human must clear)
│
└── SessionLock
    ├── acquire(timeout=60): PID check + wait loop + force-acquire on timeout
    ├── release(): delete lock file
    └── __enter__/__exit__: context manager
```

```
Watcher safety checks (per cycle):
  1. Acquire session lock (skip cycle if timeout)
  2. Check account_paused → if true, log WARNING and skip all actions
  3. After each page load: check for security challenge / CAPTCHA URL
  4. Release lock in finally block
```

```
Local handler execution order:
  1. Read rate limit state
  2. If paused or limit exceeded → leave file in Approved/ + log → return False
  3. Acquire session lock
  4. Claim file to In_Progress/local/
  5. Execute browser action
  6. Record action + release lock (finally)
  7. Move to Done/ or Rejected/
```

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LinkedIn detects automation, flags account | Medium | High | Human-like delays, conservative rate limits, session reuse, no headless mode |
| DOM changes break selectors | High | Medium | JS evaluate with 3+ fallback strategies per element; watcher skips cycle on error |
| Session lock deadlock (crash mid-session) | Low | Medium | PID-aliveness check + 60s force-acquire on timeout |
| OpenAI generates generic phrases | Medium | Low | Explicit banned-phrase list in prompt; post content always included |
| User ignores CRITICAL account_paused alert | Medium | High | `account_paused: true` blocks ALL LinkedIn execution — system self-limits |
| Connection request marked as spam | Low | High | Mandatory personalised notes, 5/day hard limit, 2nd-degree only |
